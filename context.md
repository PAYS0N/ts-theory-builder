# Steno Theory for JavaScript/TypeScript — Design Context & Handoff

> Purpose of this document: give another model **complete understanding** of an
> in-progress design conversation about building a stenography (Plover) theory
> for writing JS/TS. The project is **still in theory/decision phase** — no JSON
> dictionary and no parser have been produced yet, and won't be for a while.
> Everything below is either a locked decision, an open question, or shared
> mechanism. Do not start producing the dictionary; continue the design.

---

## 1. Project goal

Build a steno theory that makes **all core JS/TS building blocks available as
chord strokes**, so the user can write code efficiently on a steno keyboard
(Plover). Constraints and intentions:

- **Final artifact must be JSON** (a Plover dictionary). It *may* be generated by
  code, but the committed product is JSON.
- The user does **not** want strokes for single words or individual symbols —
  steno already handles those. This theory is for **multi-token structural
  blocks** (control flow, function skeletons, loops, class members, data
  structures, etc.).
- **Tabs assumed = 4 spaces initially**, but indent width must not be baked into
  cursor logic (see movement rules).
- **All cursor movement must be part of the final chord of a translation.**
  Movement is not reversible and cannot happen mid-sequence. This is the single
  most important mechanical constraint in the whole design.
- Accessible variations per **editor spec** (e.g. 2-space indent, smart-braces
  on/off, snippet-capable) should fall out of the system, not be hand-authored.
- Eventual extension to **Vim/snippet editors** with real tabstop placeholders.

## 2. Source files in the conversation (intent only, NOT parsable yet)

Two files were shared by the user. They are accessible to you on disk.

### 2a. `js.json` — the original naive/small example
A real Plover-style dictionary. Establishes the original conventions:
- Keys are chords or chord sequences split on `/`.
- Values are emitted text plus control sequences:
  - `{^}` = attach with no leading space.
  - `\{` / `\n` = literal brace / newline.
  - `{^ ^}` = a space attached both sides (used as a spacing/gap marker).
  - `{#Up}` / `{#Left}` / `{#Right}` = literal keystrokes (cursor movement).
- `SKWR` (steno "J") acts as a **namespace sigil** = "JavaScript mode."
- Right bank spells the construct; a second stroke selects a variant; trailing
  movement keys try to land the cursor at the next thing to type.
- **Known weakness that motivated the redesign:** landings were built from
  `{#Up}` + a fixed count of `{#Right}`, which is **indent-width dependent** and
  breaks under 2-space vs 4-space and across editors.

### 2b. `snippets.txt` — the extended intent file
A richer, **non-parsable** representation of intent (the user explicitly says it
is "just to indicate the general ideas," not yet machine-readable). Uses a new
template notation (see §3). Switched the namespace sigil to **`STKWR`** in this
file. Covers functions (many kinds), if/else-if families, loops, switch/case,
try/catch, array-callback methods (map/filter/reduce/forEach), declarations,
indexing, `.length`, `.at()`, console.log, throw, and a block of **type strokes**
(number/string/boolean/null/undefined/never/bigint/symbol/void/Promise/Map/Set/
Error/function/Generator) intended to be used as secondary strokes.

## 3. Template notation (the "intent" DSL in snippets.txt)

Each block is a template. The **planned** parsing pipeline (not yet built):

1. Prepend `{^}` to every value.
2. Editor-flag expansion produces variant entries / profiles:
   - `if A` → editor has **no smart braces**: close braces manually, add extra
     newlines.
   - `if O` → **don't expand `%b`**; keep as one-liner.
   - `if E` → **snippet-capable editor**: output a snippet *trigger* instead of
     literal text (triggers later mapped to actual snippets in a Vim plugin).
3. Special operators inside templates:
   - `%digit` (e.g. `%0`, `%1`) = ordered **cursor landing points**. In plain
     (non-snippet) editors the cursor only ever needs to end on `%0`. In
     snippet/Vim editors, these map to `$1…$0` tabstops.
   - `%b` = toggleable **newline-or-not** (the brace-body break).
   - `%t` = a **type**, inserted by an appended type stroke (see §6, the append
     model). Not free-typed after the fact.
   - `%d` = a `%digit` whose **count** is selected by a group of 4 binary keys
     (see §5). Produces an incrementing series of landing points.
   - `%[ … %]` = **repeat-with-separator** block (e.g. param lists, case blocks).
     Separator must be a *joiner*, not appended after the last item (no trailing
     comma). Must be defined as `%[ sep | body %]` semantics.
   - `%(expr)` = computed index (e.g. switch uses `%(2d-1)`, `%(2d)`).

**Important:** These operators are intent notation. Several inconsistencies in
the file were identified and resolved as decisions below; the file itself was
never meant to be fed to a parser as-is.

## 4. Architecture decisions — LOCKED

### 4a. Namespace
`STKWR` is the JS-mode sigil (superseding the original `SKWR` from js.json).

### 4b. Editor variation via BUILD PROFILES, not per-stroke flags
The A / E behaviors are **constant per editor**, so they become **separate
compiled dictionaries (profiles)**, and the user loads the one matching their
editor. Rationale:
- Active translation space stays at base size (you load one profile), not
  base × variants.
- **Zero per-stroke decision** about editor behavior.
- Frees the A/E vowels back into the stroke space to mean other things.
- File size / device 16 MB cap is a non-issue at this scale either way; the win
  is active-dict size and reclaimed chords, not disk.

Nuance the user raised: the *one-liner* (`O`) behavior could still be a
per-stroke option where it makes sense, but **`switch`/`case` are always
multi-line** (never collapsed). `try/catch/finally` *should* honor one-liner and
must be re-expressed with `%b` (originally it hardcoded literal tabs/`\n`).

### 4c. Movement model (indent-independent)
- Movement is **final-stroke-only** (hard constraint).
- Do **not** anchor with `{#Up}` + counted `{#Right}` (indent-dependent).
- Use **`{#End}` / `{#Home}`-relative** anchoring and count `{#Left}` backward
  from typed content. Counting Lefts relative to typed text is invariant to
  indent width.
- When a landing is on an earlier line, the indent-independent terminal pattern
  is: `{#Up}×N` (N = template line count, a fixed template constant) + `{#End}` +
  `{#Left}×K` (K = chars from line-end back to the slot, also a template
  constant). Both counts are template constants, so still indent-proof. (It is
  Up+End+Left, not Home/End alone — Home/End stay on one line.)

### 4d. Count encoding (binary key-banks)
- Counts are encoded by a group of binary keys because Plover has no live
  arithmetic — every distinct count is a literal entry.
- The **physical keys float** per construct (whichever keys are free after the
  construct is spelled): e.g. AOEU on a function's second stroke; SKWR on the
  index construct; RBGS on switch. This is forced by collision avoidance, and the
  user accepts memorizing per-construct key locations.
- **Bit-significance convention is FIXED and uniform:** **least-significant bit
  closest to the middle** of the board; since vowels sit in the middle, the
  binary **reads left-to-right**. This removes the reversed-weighting that causes
  misstrokes, even though physical keys differ per construct. Max meaningful
  count is 9.

### 4f. Declarations
Keep typed `let/const` (`let %0: %t = %1`) AND add **no-type variants**
(`const %0 = %1`) so destructuring patterns can compose onto them cleanly.

### 4g. Destructuring
Standalone pattern strokes **`[%p]`** and **`{%p}`** with an arbitrary number of
slots (via a count bank), cursor → first slot. Dropped in after a no-type
`let/const` or into a param list. (Avoid stroking a pattern into the *typed*
declaration's name slot — inline-typing a destructure is awkward.)

### 4h. Callbacks → method-selector pattern
Stop encoding *which array method* in the root chord (right bank was getting
crowded). Instead: a **method root** ("this is a member/method call") + a
**second stroke selecting the method** (map / filter / reduce / forEach / find /
findIndex / some / every / sort / flatMap …). Unifies the family under one root,
frees the right bank, and adding a method is one new second-stroke entry. 
Should be expanded to multiple method bases (ex: adding String prototype methods)

### 4i. Classes
- `constructor`, `get`, `set` are **function-shaped** → implement as
  **second-stroke variants of the function root**, reusing all param/count
  machinery.
- Methods already covered by the function/method machinery.
- Add class-creation strokes: plain `class %0 {%b%1}` and extends variant
  `class %0 extends %1 {%b%2}`, plus a small `super(%[%d, %])`.
- `static` is already a single stroke. **`this.`** to be added (cheap, frequent).
- Skipped for now (user's call): `#private` fields, get/set beyond the above.
  (A mild objection was registered that `this.`/`static` are high value; `this.`
  was accepted, static already existed.)

### 4j. Try/catch/finally
Add **`finally {%b%0}`**, a full `try/catch/finally`, and a bare `try/finally`.
Re-express try/catch with `%b` so it honors the one-liner profile.

### 4k. Data structures
Implement as **static "root + selector"** (mirrors the function/callback pattern):
a data-structure root + a second stroke choosing which structure (singly-linked
list, doubly-linked list, stack, queue, deque, min-heap, max-heap, binary tree,
BST, graph as adjacency list, hashmap, trie, …). Whole-structure static
emission; braces/newlines handled by the **profile**. These are the
highest-ROI strokes (large boilerplate). The **one** justified per-instance
modifier is **typed vs untyped** (`class Node<T>` vs `class Node`) → a
second-stroke type, not a profile. Future nicety (not v1): a standalone **node**
stroke decoupled from the operations, since several structures share a Node def.

## 5. Coverage map — what's IN vs still TO-SPEC

**Already represented in intent files / agreed:** function family
(regular/arrow/async/async-arrow/generator/IIFE/method), if & else-if families
(incl. `=== false/undefined/null`, `=== %1`), else, loops
(count-up/count-down/C-style/for-of/for-in/while/do-while), switch/case(±default),
try/catch (+throw), array callbacks (map/filter/reduce/forEach with expr & block
variants), indexing `[]`/`[%d]`/`.at()`, `.length`, `console.log`, typed/no-type
let/const, the type-stroke set.

**Agreed to ADD (still to spec in stroke notation):**
- Objects / destructuring: object literal (shorthand + computed keys), object &
  array destructuring patterns `{%p}`/`[%p]`, destructuring params.
- Classes: creation, extends, super, constructor/get/set (as function variants),
  `this.`.
- Callback method-root family expanded (find/findIndex/some/every/sort/flatMap…).
- `finally`, full try/catch/finally, bare try/finally.
- Data structures (root + selector set above).
- Ternary skeleton `%0 ? %1 : %2` and template literal `` `${%0}` `` (optional;
  value is only the landing since they're symbol-light).

**Deferred by user (too new to design well right now): MODULES and ASYNC/Promise
detail.** Modules (import named/default/`* as`/side-effect, export const/function/
default, re-export) and richer async (`new Promise`, async arrow, `.then/.catch`
chains, `Promise.all/race/allSettled`, `for await…of`) were identified as core
and high-frequency, but the user asked to **skip them for now**. Flag them as the
top items to revisit.

## 6. Parked / future capabilities

- **Capture-last-value:** the user can (regardless of editor) capture the most
  recently typed token and re-emit it inside a stroke's output. Powerful (wrap
  last value, swap, declare-from-expression) but **hard and explicitly out of
  scope for now**. Design should leave room for it rather than precluding it.
- **Snippet/Vim profile:** the `E`-flag path emits snippet *triggers*; a later
  Vim plugin maps those triggers to real snippets whose `$1…$0` tabstops give
  indent-aware, editor-computed placeholders — the proper home for nested
  container types and multi-field templates that plain steno can only
  single-land.
- **Standalone `Node` stroke** for shared data-structure node defs.
- **Modules & async/Promise** construct sets (deferred; top of the revisit list).

## 8. Working-style constraints (from user preferences — apply these)

- **Diagnose before fixing.** If something looks wrong, first design how to
  *confirm* it's the cause, then how to fix — don't jump to a patched line.
  (Example given: for `sum(a,b){return a-b}`, first check whether `sum(2,3)===5`,
  don't immediately rewrite to `a+b`.)
- **Plan and confirm before writing code.**
